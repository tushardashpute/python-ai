Real -world flow 

1. take the pdf or document 
2. extract the text
3. split it into smaller chunks
4. create embeddings out of these chunks 
5. Store these embedings in a vector DB 
6. when a user now asks a q, embed the question as well 
7. find the closest chunks by semantic similarity
8. send those chunks to the LLM 
9. LLM answers using that retrieved knowledge

PDF -> text -> chunks -> embeddings -> retrieval -> LLM -> output 

this is your actual RAG pipeline in production .

internally how it works, top-k retrieval

Step 1 - Search across documents (chunks)
eg
Doc 1: Refunds .....
Doc 2: Shipping .....
Doc 3: Damaged .....

Step 2 - Score Relevance 
Refund -> relevant
Damaged -> relevant
Shipping  -> not relevant.
CEO - > not relevant 
returns - > relevant

Step 3- Top-k retrieval happens 
eg k=2 
Top 2
[Refund Policy]
[Damage Policy]


step 4
Combine into one context 
Context now becomes:
    "Refunds are allowed within 30 days",
    "Damaged items can be replaced",

Step 5 - now send this to LLM
LLM now answers using BOTH peices of info. 

[user Question] -> Search across Chunks -> Rank by relevance -> [top-k selected =2 in our case] (k is usually 3-5 in real systems) -> LLM --> Answer/Output.


choosing top-k is a trade-off too:
Too small -> can miss info
too large -> add noise and more tokens 


top-k => how much context you allow into the model to process. 

Case k=1 - too small, only refunds doc/chunk will be fed so LLM will miss the damaged item value
it can look like its hallucinating.

k=2 it was good

k=3 or k=4 (too large)
REfund + damage + shipping + discounts + CEO etc 
Problem:
    - noise
    - confusion 
    - higher cost (tokens)

small K             Large K 
Fast                Expensive 
Focused             Noisy 
Migh miss info      might confuse the model - increate cost 

k is not a random choice - its a trade-off that the engineer makes wrt their system


this is something to remmeber when designing the system
